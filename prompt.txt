CONTEXT AND PROBLEM STATEMENT

My physical volume buttons are broken on my Android phone. I cannot take screenshots or control volume without them. I need a floating button on my screen — permanently visible on top of all other apps — that gives me volume up, volume down, and screenshot functionality from anywhere on the phone.

The phone I am building this for is a Walton Primo RX7 Mini. It runs Android 9.0 Pie which is API level 28. It has 3 GB of RAM and a MediaTek Helio A22 processor. This is the most important fact about this project: the phone is old and low on RAM, and its operating system aggressively kills apps running in the background. Every app I have tried from the Play Store disappears within minutes because the OS kills it under memory pressure. This app must not die. Building an app that survives permanently in the background on this device is the number one engineering priority. Everything else is secondary to this.

The app is called FloatKey.

WHAT THE APP MUST DO

When FloatKey is running, a small floating circular button must be visible on the screen at all times, even when other apps are open, even when the phone is idle, even after the phone is rebooted. This bubble must sit on the edge of the screen. When the user is not interacting with it, the bubble must fade to about 18 percent opacity so it does not distract from whatever the user is doing. The moment the user touches it, it comes back to full opacity.

When the user taps the bubble, a small panel of three buttons appears next to it. These three buttons are Volume Up, Volume Down, and Screenshot. Volume Up increases the current volume by one step and shows the native Android volume slider on screen. Volume Down decreases the volume by one step and shows the same slider. The Screenshot button captures the entire screen and saves it to the device gallery. The bubble and panel must be completely hidden and invisible during the screenshot capture so they do not appear in the saved image.

The user can drag the bubble to any position on the screen. When they let go, the bubble must snap to the nearest left or right edge of the screen with a smooth animation. The bubble must remember where it was last placed and return to that position after a reboot.

TECH STACK — FOLLOW THIS EXACTLY AND DO NOT DEVIATE

Language: Kotlin only. No Java anywhere.

Minimum SDK: 28. Target SDK: 34. Compile SDK: 34.

Third-party libraries: absolutely zero. No Retrofit, no Dagger, no Hilt, no Glide, no Coil, no Coroutines library beyond what ships with Kotlin stdlib, no analytics SDK, no crash reporting SDK, no ads SDK, nothing. The only allowed dependencies in build.gradle are core-ktx and appcompat, and only if they are genuinely needed. Every external dependency increases APK size and RAM footprint, which this device cannot afford.

Build configuration: enable minifyEnabled true and shrinkResources true for the release build type. This is mandatory to keep the APK under 2.5 megabytes.

The UI must use custom Views drawn with Canvas. Do not use Jetpack Compose. Do not inflate heavy XML layouts for the floating views. Keep it programmatic and minimal.

The floating overlay must use WindowManager with the layout type TYPE_APPLICATION_OVERLAY. This is the only correct approach for an overlay that sits above all other apps.

The service must be a Foreground Service. This is non-negotiable. A background service will be killed by the OS on this device within minutes. A foreground service with an active notification is protected by Android and will not be killed under normal memory pressure.

For screenshot capture on API 28, use the MediaProjection API with an ImageReader and a VirtualDisplay. Do not attempt to use AccessibilityService.takeScreenshot() because that method only exists from API 30 onwards and will crash on this device.

For volume control, use AudioManager.adjustVolume() with the ADJUST_RAISE or ADJUST_LOWER flag combined with FLAG_SHOW_UI so the native volume panel appears.

PROJECT STRUCTURE — CREATE EXACTLY THESE FILES AND NO OTHERS

The package name is com.floatkey.

The Kotlin source files you need to create are:

LauncherActivity.kt — this is the app entry point. It has no layout file. Its only job is to check if the SYSTEM_ALERT_WINDOW permission is granted. If it is not granted, it must redirect the user to the system settings screen where they can grant it. If the permission is already granted, it starts the FloatKeyService as a foreground service and immediately calls finish() on itself. The Activity must not stay in the task stack after it has done its job.

FloatKeyService.kt — this is the heart of the application. It extends Service. In onStartCommand it must return START_STICKY. The very first operation in onStartCommand, before anything else, must be to create a notification channel and call startForeground() with a persistent notification. This is critical. If startForeground is not called immediately, Android can kill the service before it has time to set up. After calling startForeground, it gets the WindowManager system service, creates a BubbleView, creates an ActionPanelView, and adds both to the window using windowManager.addView(). It manages the lifecycle of these views and provides callbacks between them and the ScreenshotManager. In onDestroy it removes all views from the WindowManager and calls stopForeground.

BubbleView.kt — extends View. This is the draggable floating circle. It draws itself using canvas.drawCircle() with a Paint object. No bitmap assets. It handles touch events: ACTION_DOWN resets the idle timer and brings opacity to full, ACTION_MOVE updates the WindowManager layout params to move the bubble following the finger, ACTION_UP determines whether the gesture was a tap or a drag. If the total movement was less than 8dp it is a tap and it must call a listener to toggle the action panel. If it was a drag it must snap the bubble to the nearest screen edge using a ValueAnimator with an OvershootInterpolator for a spring-like feel. It must also implement the idle fade: a Handler posts a Runnable with a 3000 millisecond delay. When that Runnable fires it animates the view alpha from 1.0 to 0.18. Any touch event cancels the pending Runnable and resets it.

ActionPanelView.kt — this is the panel of three buttons. It is a vertical LinearLayout-style arrangement created programmatically. It contains three ImageButton or equivalent views for Volume Up, Volume Down, and Screenshot. It has a show() method that animates the panel into visibility using scale from 0.7 to 1.0 combined with alpha from 0 to 1 over 150 milliseconds with a DecelerateInterpolator. It has a hide() method that reverses this over 120 milliseconds. The buttons must call back to the FloatKeyService when tapped. The background must be a dark semi-transparent rounded rectangle, color approximately #1E1E2E at 95 percent opacity.

ScreenshotManager.kt — manages the entire screenshot lifecycle. It holds a reference to the MediaProjection object which it caches after the first permission grant and reuses for all subsequent screenshots. To take a screenshot: first it hides both the bubble and the panel by setting their alpha to 0 instantly with no animation, then it posts a 100 millisecond delay on a Handler before triggering the capture to ensure the views are fully gone from the rendered frame, then it acquires the latest Image from the ImageReader, reads the byte buffer, creates a Bitmap, releases the Image immediately, saves the Bitmap to MediaStore under Pictures/Screenshots as a PNG with a timestamp filename, then restores the bubble and panel visibility, and finally shows a Toast confirming the save. The Bitmap creation and file writing must happen on a background thread. Only the Toast and view restoration happen on the main thread.

BootReceiver.kt — extends BroadcastReceiver. In onReceive it checks that the action is ACTION_BOOT_COMPLETED and then calls context.startForegroundService() with an Intent targeting FloatKeyService. This ensures the floating button comes back after every reboot without the user needing to open the app.

THE ANDROID MANIFEST — FOLLOW THIS STRUCTURE PRECISELY

Declare these permissions:

android.permission.FOREGROUND_SERVICE — this is a normal permission, granted automatically at install.

android.permission.RECEIVE_BOOT_COMPLETED — this is a normal permission, granted automatically at install.

android.permission.WRITE_EXTERNAL_STORAGE — this is a dangerous permission, must be requested at runtime when the user first tries to take a screenshot.

android.permission.SYSTEM_ALERT_WINDOW — this is a special permission that cannot be requested through the normal runtime flow. The user must be sent to the system settings page to grant it. The LauncherActivity handles this check and redirect.

Declare the LauncherActivity with the MAIN action and LAUNCHER category intent filter, and android:exported="true".

Declare a second Activity called ScreenCaptureRequestActivity with android:exported="false" and android:theme set to a translucent no-title-bar theme. This Activity exists solely to show the MediaProjection permission dialog, which Android requires to be requested from an Activity context and cannot be requested directly from a Service. This transparent Activity is launched by FloatKeyService when a screenshot is requested and the MediaProjection token has not yet been obtained. It requests the capture intent, receives the result in onActivityResult, passes the resultCode and data back to the service, and immediately calls finish() on itself.

Declare FloatKeyService with android:exported="false" and android:foregroundServiceType="mediaProjection".

Declare BootReceiver with android:exported="true" and an intent-filter for android.intent.action.BOOT_COMPLETED.

CRITICAL IMPLEMENTATION DETAILS YOU MUST GET RIGHT

The most common reason floating overlay apps die on low-RAM devices is that the developer uses a background service instead of a foreground service, or they call startForeground() too late. In FloatKeyService, startForeground() must be the very first call inside onStartCommand before the WindowManager setup, before creating views, before anything.

The START_STICKY return value in onStartCommand is mandatory. This tells the Android system that if it ever does kill the service under extreme memory pressure, it should automatically restart it as soon as memory is available again.

The notification used for startForeground must be created on a notification channel with IMPORTANCE_LOW. Do not use IMPORTANCE_MIN because on some Android 9 devices that causes the notification to be silently dropped, which can cause startForeground to fail. IMPORTANCE_LOW shows the notification silently without sound or vibration. The notification text should be "FloatKey is running" and it must include one action button labeled "Stop" that sends a broadcast to the service telling it to call stopSelf().

The WindowManager LayoutParams for the bubble and panel must include the flag FLAG_NOT_FOCUSABLE. This is critical. Without it, the overlay will steal keyboard focus from other apps, breaking text input in every app on the phone while FloatKey is active. Also include FLAG_LAYOUT_IN_SCREEN. The pixel format must be PixelFormat.TRANSLUCENT to support transparency.

For the bubble position persistence, use SharedPreferences to save the X and Y coordinates every time the bubble snaps to an edge after being dragged. On service start, read these values and set the initial position of the bubble to the saved coordinates. If no saved coordinates exist, default to the right edge at 40 percent from the top of the screen.

To measure screen dimensions for positioning, use the WindowManager display metrics. Do not use Resources.displayMetrics because on some devices with cutouts or navigation bars it gives incorrect values for overlay positioning.

When dismissing the action panel when the user taps outside it, you can add a full-screen transparent WindowManager view at a lower z-order than the panel. When this transparent overlay receives a touch event, it dismisses the panel and removes itself from the WindowManager.

For the edge snap animation, use a ValueAnimator on the X coordinate of the WindowManager LayoutParams. Inside the animator's update listener, call windowManager.updateViewLayout() on every frame. The duration should be 200 milliseconds with an OvershootInterpolator with a tension value of 1.2 to give a subtle spring bounce.

The long press behavior on the volume buttons is implemented with a Handler and a Runnable. When the user presses and holds the button, start posting a Runnable every 300 milliseconds that calls adjustVolume(). When the user lifts their finger, remove the callbacks.

For saving the screenshot to the gallery, use MediaStore.Images.Media.insertImage() or ContentValues with MediaStore.Images.Media.EXTERNAL_CONTENT_URI. Set the MIME type to image/png, the RELATIVE_PATH to Pictures/Screenshots, and the DISPLAY_NAME to FloatKey_screenshot_ followed by the current timestamp in milliseconds. On API 29 and above you would also set IS_PENDING to 1 during write and then 0 after, but since the minimum SDK is 28 you should use the simpler ContentValues approach that works on API 28.

VISUAL DESIGN SPECIFICATION

The bubble is a filled circle, 52dp in diameter. The fill color is #1A1A2E. It has a thin inner ring, 2dp wide, white at 30 percent opacity, drawn just inside the circumference. The icon drawn inside is a 3x3 grid of 9 small dots in white at 60 percent opacity. Draw this with canvas.drawCircle() for the dots — 9 circles of about 3dp radius each, evenly spaced in a grid pattern centered in the bubble. The idle opacity is 0.18 and the active opacity is 0.92. The transition between idle and active is a simple ObjectAnimator on the alpha property over 300 milliseconds.

The action panel has a dark rounded rectangle background, color #1E1E2E, 95 percent opacity, with 16dp corner radius. It contains three buttons stacked vertically with 8dp spacing between them. Each button is 44dp by 44dp. The icons are vector drawables: a volume-up speaker icon, a volume-down speaker icon, and a camera icon. Icon color is white at 85 percent opacity. The pressed state color for icons is #3D8BFF. There are no labels or text on the buttons, only icons.

The notification is plain text only. No large icon, no big picture, no custom layout. Keep it minimal so it takes up the least possible space in the notification shade.
