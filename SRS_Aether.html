1 Introduction

1.1 Purpose

This document defines the complete software requirements for Aether — a minimal Android application that presents a persistent floating overlay button, giving the user fast access to volume control and screenshots when the physical volume buttons are unavailable.

1.2 Background & Problem Statement

The user's Walton Primo RX7 Mini has a broken physical volume rocker. Without it, adjusting media/call volume requires diving into Settings, and taking screenshots (which normally requires Power + Volume-Down simultaneously) is entirely impossible.

Existing solutions from the Play Store fail because they are heavy — they load ads, analytics SDKs, large UI frameworks, and background workers that the device's aggressive memory manager kills under low-RAM pressure. The goal is to build a first-party, zero-bloat solution that the OS cannot justify killing.

1.3 Scope

Aether is a single Android application (.apk) containing:

  • IN-SCOPEA draggable floating bubble rendered on top of all other apps
  • IN-SCOPEVolume Up / Volume Down controls
  • IN-SCOPEScreenshot capture and immediate save to Gallery
  • IN-SCOPEA foreground service that keeps the overlay alive at all times
  • OUTCloud sync, user accounts, ads, analytics, crash reporting SDKs
  • OUTSupport for Android versions below 9.0 (API 28)

2.1 Why Services Get Killed on This Device

Android 9 introduced stricter background execution limits. On a 3 GB device with stock Android Pie, the OS's Low Memory Killer (LMK) aggressively targets background services. A background service with no foreground notification is classified as "cached" — first in line to be killed under any memory pressure. The only reliable defense is a Foreground Service with an active persistent notification, which the OS will not kill unless the system is in extreme distress.

Service TypeLMK PriorityWill Survive?
Background Service (no notification)Cached — killed first❌ No
Foreground Service (with notification)Visible — protected✅ Yes
System / Bound Service to systemSystem — never killed✅ Yes (not available to 3rd party)

3 Product Overview

Aether launches as a small floating circle (the "Bubble") on the screen edge. When idle, it fades to low opacity so it does not distract. A single tap expands a minimal action panel with three buttons: Volume Up, Volume Down, and Screenshot. All actions execute instantly and the panel collapses back to the bubble. The bubble is draggable to any screen position and snaps to the nearest edge on release (edge-snap UX, identical to iPhone's AssistiveTouch).

✅
Design Philosophy

No animations that stall the main thread. No background workers beyond the single foreground service. No third-party SDKs. Every millisecond of CPU time this app uses is a millisecond taken from the user's other apps.

4 Functional Requirements

4.1 Floating Overlay

  • FR-01MUST The app must display a circular floating bubble permanently on-screen, above all other apps and system UI, using WindowManager with type TYPE_APPLICATION_OVERLAY.
  • FR-02MUST The bubble must be draggable by touch to any position on screen.
  • FR-03MUST On drag-release, the bubble must snap to the left or right screen edge (whichever is nearer) with a smooth spring animation.
  • FR-04MUST When no interaction has occurred for 3 seconds, the bubble must fade to 18% opacity. It must return to full opacity on touch.
  • FR-05MUST A single tap (distinguished from a drag by a movement threshold of < 8dp) must toggle the action panel.
  • FR-06SHOULD The action panel must appear anchored near the bubble with a fade + scale animation of ≤ 150ms duration.
  • FR-07SHOULD Tapping anywhere outside the action panel must dismiss it.

4.2 Volume Control

  • FR-08MUST The Volume Up button must call AudioManager.adjustVolume(ADJUST_RAISE, FLAG_SHOW_UI) on the current active audio stream.
  • FR-09MUST The Volume Down button must call AudioManager.adjustVolume(ADJUST_LOWER, FLAG_SHOW_UI) on the current active audio stream.
  • FR-10MUST The native Android volume panel (OSD) must appear when volume changes, using FLAG_SHOW_UI, giving visual feedback identical to the physical button experience.
  • FR-11COULD Long-pressing Volume Up / Down buttons may rapidly repeat volume changes at 300ms intervals for fast adjustment.

4.3 Screenshot Capture

  • FR-12MUST The Screenshot button must capture the full screen contents as a PNG image.
  • FR-13MUST The screenshot must be saved to the device's public Pictures/Screenshots directory, accessible from the Gallery app.
  • FR-14MUST The Aether overlay (bubble + panel) must be hidden/invisible during capture so it does not appear in the screenshot.
  • FR-15MUST On Android 9 (API 28), screenshot must use the MediaProjection API. The projection token must be obtained once per session and reused to avoid repeated permission prompts.
  • FR-16MUST After a successful capture, show a small toast or subtle notification confirming the save.
  • FR-17SHOULD If the MediaProjection permission has not been granted yet, automatically launch the permission request dialog before the first screenshot.
ℹ️
Screenshot Note — Android API 28

AccessibilityService.takeScreenshot() is API 30+ and is NOT available on this device. The only correct approach on API 28 is MediaProjectionManager.createScreenCaptureIntent() combined with ImageReader and VirtualDisplay. The token must be cached after first grant to avoid showing the system dialog on every screenshot.

4.4 Service Persistence

  • FR-18MUST The overlay must be managed by a Foreground Service with a permanent, low-priority notification.
  • FR-19MUST The service must use START_STICKY so Android automatically restarts it if it is ever killed by the OS.
  • FR-20MUST A BroadcastReceiver must listen for BOOT_COMPLETED to relaunch the service on device restart.
  • FR-21SHOULD The service's notification must show a single "Stop Aether" action so the user can cleanly exit if desired.

4.5 Permissions Onboarding

  • FR-22MUST On first launch, the app must detect if SYSTEM_ALERT_WINDOW overlay permission is not granted and immediately redirect the user to the system settings screen to grant it.
  • FR-23MUST The main launcher Activity must finish (close) after launching the service, leaving zero UI in the task stack.
  • FR-24MUST On return from permissions screen, the app must automatically start the service if all required permissions are satisfied.

7.3 Notification

The persistent foreground notification must be minimized / silent: no sound, no vibration, IMPORTANCE_MIN or IMPORTANCE_LOW. Text: "Aether is running." with a "Stop" action button. This notification is a technical requirement to keep the service alive — it should be as unobtrusive as possible.
